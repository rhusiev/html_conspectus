variant - Voluntary Engagements

Initial description

The domain of voluntary engagements includes volunteers doing work for organizations, individuals or being on their own.

They do it for some time periods, can do fundraising from individuals (though they have limited resources) or gain grants from organizations. Some can also do work for stypendiyna hodynas because of the need to get a scholarship in Universities. However, every volunteer has limited time they can spend on volunteering.

Works consist of some tasks, which are done at some location

Modelling

Requirements

Voluntary work is a work that is done for free and consists of:

  • The Voluntary Work:
    • Is done by a volunteer
    • It is done for some employer (which can be an organization, an individual or the volunteer themselves)
    • It is done over some periods of time
    • It can also be done by fundraising:
      • from other people (each having some money they are willing to give)
      • from organizations
    • Sometimes for stypendiyna hodynas
      • for some university
      • Takes some time and money
      • Consists of some tasks, each of which is done at some location (or online)

Primary Entity Type

The primary entity type is Work, as everything revolves around it.

Then, I introduce all the possible attributes of different possible future entities

The TimePeriod is composed of two things: TimeStart and TimeEnd. Therefore, it will be a composite attribute. There also may be a few different TimePeriods, so it will be a multivalued one.

TimeToSpend is calculated from all the TimePeriods, therefore it is a computed attribute.

At this point, I also decide to introduce a hint section to elaborate what I mean by different notation. I will expand it in the future, if needed.

Defining more enitity types

I split the attributes into different entity types.

The six main ones are:

  • VoluntaryWork
  • University
  • Volunteer
  • Individual
  • Employer
  • Task

Distribute Attributes

Now I'll draw them separately. I also add an "Organization" entity type. It does not contain any special attributes, but is crucial for future relationships

Basic Relationships

Next, I specify the relationships between the entities. As aVolunteer doing VoluntaryWork for an Employer needs three entities to happen, I'll introduce a ternary relationship.

Foreign Keys

To have Tasks uniquely identified, I'll use the foreign key of WorkId (from VoluntaryWork) and its own one TaskNo

Composites and Aggregates

Now, I'll specify the multiplicities of the relationships to make the model more precise with the help of composites and aggregates.

Most relationships are unrestricted, but some are not:

  • VoluntaryWork has to have some Tasks (and a Task has to be a part of some work)
  • Doing a voluntary work requires at least a volunteer, some employer (can be volunteers themselves) and the work to do

At this point I will also mark Volunteer and Task as weak entities. The reason is that a Volunteer is no longer a volunteer without some VoluntaryWork, and the task is a part of some work.

Generalizations

There are different groups that can be employers:

  • Organizations
  • Individuals (a person a volunteer is helping)
  • Or volunteers themselves (e.g. doing some work by themselves)

Results

In the end, I have a model that is generally precise for the task of representing the relationships when doing some voluntary work. It can be used in the future to create a database for a management system or for other tasks that require prior modelling.

We can successfully model people volunteering for other people, some organizations or doing some work on their own for everyone's good. We can also model the funding of a volunteer, as well as some motivations that can stand behind it (stypendiyna hodynas). In the end, the model also gives some grasp on the tasks that are done during the work and the time spent on them.

For a management system all the necessary information and relationships are present.
